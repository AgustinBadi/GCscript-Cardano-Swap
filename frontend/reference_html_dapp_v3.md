# GameChanger UDC / GCScript Boilerplate — Technical Specification

## 1. Purpose

This document specifies the current official single-file vanilla JavaScript boilerplate for Cardano dapps built on the GameChanger Universal Dapp Connector (UDC) and GCScript DSL.

The boilerplate is the canonical HTML runtime target for GameChanger code generation tooling. Its job is to transform one or more GCScript intent templates into a working end-user dapp with minimal manual coding, while preserving a strong separation between:

- developer-customizable use-case code
- generic runtime/tooling infrastructure
- end-user-facing UI behavior

The current boilerplate combines these concerns into one compact artifact:

- intent template storage
- automatic field generation from intent `args`
- persistent shared app state
- UDC URL and QR generation
- wallet result capture and decode
- optional connected-wallet UI patterns
- optional validation, migration, and air-gapped response flows

## 2. Scope

This specification covers the readable, non-extreme low-footprint browser edition of the boilerplate.

This edition:

- imports the official `@gamechanger-finance/gc` browser bundle
- uses browser storage and browser DOM APIs
- exposes one global reactive app state proxy
- remains suitable as the reference target for code generators, examples, and wallet-integrated flows

This specification does not define the future zero-dependency transport implementation in full detail. However, the current architecture is explicitly designed so that a reduced-footprint edition can be derived by:

- removing developer-facing comments and documentation
- swapping the external GC browser bundle for local encode/decode shims

The conceptual runtime model must remain the same across both editions.

## 3. Product Positioning and Toolchain Role

### 3.1 Canonical generated HTML target

This boilerplate is the canonical generated HTML dapp form factor for GameChanger UDC integrations. It is intended primarily to be emitted by tools, not hand-authored from scratch in most cases.

### 3.2 Intended producer tools

The intended producer tools are:

- the official GC library and CLI
- the GameChanger wallet where wallet-driven generation or reuse is appropriate
- future code generators that synthesize dapp frontends from GCScript intent definitions

### 3.3 Intended consumer developers

The primary developer audience is broad by design:

- junior Cardano developers onboarding into GCScript and UDC
- non-specialist web developers who need a safe default runtime shell
- advanced teams that want a concise generated starting point that they can selectively extend

The boilerplate therefore prioritizes:

- a recognizable lifecycle
- narrow customization surfaces
- stable DOM ids and classes
- readable inline documentation
- minimal gotchas between state, business logic, and rendering

## 4. Core Design Goals

### 4.1 Near-zero manual coding

A developer should be able to define one or more intents and obtain a functional dapp without manually building the form layer, the persistence layer, or the encode/decode bridge.

### 4.2 Shared persistent multi-intent state

The default runtime must be able to hold multiple intents at once, preserving:

- mutable args
- intent-local exports
- shared exports
- auxiliary custom state
- UI and options state

### 4.3 End-user-first UX

Generated dapps are for end users first. The default visible UI must remain minimal, task-focused, and action-oriented. Developer-facing explanations should remain behind options, debug flows, or inline code documentation rather than cluttering the primary UX.

### 4.4 Ultra-concise delivery model

The boilerplate must remain small enough to be practical as a single HTML file and economically realistic for byte-sensitive deployment targets such as GCFS.

### 4.5 Strong separation of concerns

Use-case logic must live in the developer customization section.
The advanced autogenerated runtime must remain intent-agnostic and use-case-agnostic.

### 4.6 Stable styling and DOM targeting surface

Generated DOM must remain cleanly targetable by external CSS and downstream theming through stable ids and classes.

### 4.7 Ultra-simple lifecycle

The lifecycle visible to developers must remain conceptually simple:

1. create app
2. run logic
3. render UI

The boilerplate formalizes this as `createApp({ config })`, `update(app)`, `logic(app)`, and `render(app)`.

### 4.8 Single-source-of-truth configuration

All default dapp initialization data must be discoverable in one primary configuration object inside the global app instance.

### 4.9 Reduced-footprint compatibility

The readable reference edition and the reduced-footprint on-chain edition must share the same architecture and behavioral model.

### 4.10 Human-friendly API surface

Helpers and methods should use object arguments when multiple inputs are needed, with self-descriptive naming.

## 5. Design Philosophy and Patterns

### 5.1 One global reactive app instance

The current boilerplate uses a single global reactive proxy:

```js
const app = createApp({
  config: {
    // defaults, fields, intents, logic, render, bind
  }
});
```

This makes the app state, behavior, and customization model visible from one entry point.

### 5.2 Two-layer architecture

The boilerplate is intentionally split into two large conceptual zones:

#### Developer Customization Section

This section contains use-case code and should be the primary edit surface for generated dapps.

It defines:

- defaults
- field schema
- read-only intent templates
- example wallet normalization
- example intent resolution
- business rules in `logic(app)`
- custom UI rendering in `render(app)`
- custom event bindings in `bind(app)`

#### Advanced Autogenerated Runtime

This section contains shared infrastructure and should usually remain unchanged.

It provides:

- state persistence
- proxy-based reactivity
- generic field inference
- generic form rendering
- URL and QR generation
- wallet result decoding and merge behavior
- migration handling
- lifecycle orchestration

### 5.3 Generic runtime, specific customization

The runtime is designed around a core philosophy:

- the runtime provides generic mechanics
- the customization section provides current use-case meaning

That pattern allows the same runtime to keep working even if the example intents are replaced with one entirely different intent set.

### 5.4 Default vs dynamic intent separation

The boilerplate maintains a strict distinction between:

- `app.defaultIntents`: read-only intent templates
- `app.intents`: live editable runtime copies

This supports:

- reset behavior
- code generation
- predictable state mutation
- cascade updates when intent templates change

### 5.5 Config-driven field abstraction

The current field-generation layer is driven by `app.config.ui.currentFields` plus raw JSON inference. This means:

- a form can be generated automatically from intent args or options JSON
- developers can override specific fields declaratively
- custom buttons and custom HTML blocks can be inserted through the same schema
- the same abstraction can be reused in React and NodeJS renderers

### 5.6 Event-like transient wallet results

Wallet return data is treated as a transient event source, not permanent derived state. The decoded result is consumed into persistent state and then cleared, preventing stale reprocessing loops.

### 6.2 Current runtime model

The current runtime is centered on:

- `createApp({ config })`
- `update(app)`
- `logic(app)`
- `render(app)`
- a proxy-based state mutation model that schedules updates automatically

### 6.3 Current application shell

The validated HTML shell currently contains:

- hero/header with app title and description

- migration banner card before the main action card
- main action card with intent selection, summary, details, actions, statuses, and options panel
- optional debug sidebar
- footer with community, docs, website, and GC library links

## 8. Configuration and Customization Model

### 8.1 Global configuration object

The current boilerplate initializes the app through a global `createApp({ config })` call, not through a separate exported constant. This is now the primary source-of-truth pattern. fileciteturn24file12

### 8.2 Defaults

`config.defaults` provides the initial value source for:

- `options`
- `custom`

These defaults also act as the reset source.

### 8.3 Field schema

`config.ui.currentFields` is the declarative override surface for generated fields. It supports:

- `select`
- `checkbox`
- `text`
- `number`
- `textarea`
- `button`
- custom HTML-like use cases through field metadata

The current implementation includes option-level fields and intent-specific fields, including an example manual response decode control in the options section. fileciteturn24file4

### 8.4 Intent templates

`config.defaultIntents` stores read-only GCScript templates. The current reference boilerplate still includes three example intents:

- `connect`
- `payment`
- `stakeDelegation`

These remain reference examples, not mandatory business requirements of the runtime.

### 8.5 Custom helpers in the customization section

The current customization section contains example use-case helpers such as:

- `normalizeConnectedWallet({ decoded, value })`
- `getIntentCode({ app, intentKey, code })`
- `logic(app)`
- `render(app)`
- `bind(app)`

This pattern formalizes the philosophy that intent-specific behavior belongs in the customization section, not inside the generic runtime.

## 9. Lifecycle Contract

### 9.1 Entry point

The app starts by constructing the global reactive app proxy through `createApp({ config })` and then running the update cycle.

### 9.2 Update cycle

The update cycle is responsible for:

- running business logic
- rebuilding derived UI state
- rendering the DOM
- persisting the latest state where appropriate

### 9.3 Logic phase

`logic(app)` is the primary place for deriving:

- UI visibility flags
- field lists
- validation messages
- action disabled state
- intent summary state
- custom use-case state from transient results

### 9.4 Render phase

The runtime render phase is generic, while `config.render(app)` allows additional custom UI behaviors to be layered on top.

### 9.5 Bind phase

The runtime handles generic bindings, while `config.bind(app)` adds custom bindings such as the connected-wallet widget actions.

## 10. UDC Integration Contract

### 10.1 Browser GC bundle

The current boilerplate imports the official GC browser bundle and expects the browser global GC API to provide:

- URL generation
- QR generation
- wallet result decoding

### 10.2 URL generation

The runtime builds an intent URL from the resolved intent code and current transport options.

### 10.3 QR generation

The runtime builds QR data from the resolved intent code and current transport options. The current boilerplate includes an explicit QR action button and generates QR output as PNG by default. The QR action is shown only when QR support is enabled and the current action is available. fileciteturn24file19

### 10.4 Wallet result decode

Wallet results can currently enter the runtime through two paths:

- normal redirect return handling from the browser URL
- manual pasted response handling from the options panel

The manual path exists specifically to support air-gapped and QR-based workflows. The current field schema includes a `manualResponse` textarea plus a `manualResponseSubmit` action that parses a wallet return URL, extracts the `result` parameter, decodes it, merges it into state, and then clears the input on success. fileciteturn24file4

### 10.5 Return URL constraint

The runtime must not encode implementation-control query strings into the intent return URL. Return handling remains a local runtime responsibility after the wallet response is captured.

## 11. Action System

### 11.1 Action section

The current action system is grouped in `#actions-section` and can be disabled entirely through `useActionSection`. The runtime derives `app.ui.actionSectionDisabled` from that flag and uses it to hide:

- the action controls
- action status
- field status

### 11.2 Current action controls

The current validated action set includes:

- reset intent args
- toggle options
- copy link
- open QR
- continue/open intent

### 11.3 Copy link

The copy-link action is part of the general action section and is hidden or disabled when action-state rules require it.

### 11.4 QR action

The QR action uses a link-based model similar to the main open-intent action:

- it stores a generated QR data URI in the href
- it opens in a new tab by default
- it attempts popup-like behavior through click handling
- it updates action status on success or failure
- it is controlled by `useQR`, `useAction`, `useActionSection`, current action availability, and action validation state

### 11.5 Primary continue action

The `connect-link` control is the primary action for the current intent and is rendered on its own row below the utility buttons. The layout fix is part of the validated current UI shell.

## 12. Validation and Status Pattern

### 12.1 Current philosophy

Validation lives in the developer customization section, typically inside `logic(app)`, so that developers can express domain rules in the same place where they already derive business state.

### 12.2 Current reference validation examples

The latest reference boilerplate includes examples such as:

- blocking error when payment quantity is not greater than zero
- blocking error when address prefix does not match the selected Cardano network
- warning when return URL does not use HTTPS
- warning when stake delegation ticker is unusually short

These are examples of pattern and placement, not universal business rules.

### 12.3 Status channels

The runtime currently uses:

- `field-status` for field- and validation-related messages
- `action-status` for action execution results and transport feedback

### 12.4 Tone system

The current style layer defines a reusable tone system for both themes:

- info
- success
- warn
- danger
- mute

This tone system is consumed by:

- status labels
- migration card styling
- tone helpers and future extensible message surfaces

## 13. Migration and Persistence Pattern

### 13.1 Version-aware persistence

The runtime compares the persisted app version with the current config version. When they differ, the runtime enters a migration-needed state.

### 13.2 Migration banner

The current implementation renders a dedicated warning card before the main action card when migration is required. This card informs the user that saved local data differs from the current app version and that updating may replace older saved information. The migration banner is now a first-class UI section, not an inline sub-block. fileciteturn24file11

### 13.3 Manual migration action

The user can explicitly choose to update local persisted state to the latest app version. While migration is pending, action execution is blocked.

### 13.4 Persistence rule during migration

The runtime avoids normal persistence writes while migration is pending, preventing accidental overwrite of legacy stored data before the user confirms the update.

## 14. Connected Wallet Widget Pattern

### 14.1 Purpose

The connected-wallet widget is a documented example of how developers can add custom UI that consumes shared app state.

### 14.2 Enablement rule

The current example enables the widget only when:

- `useConnect` is true
- a `connect` intent exists in `defaultIntents`

This keeps the example safe when intent sets change.

### 14.3 Behavior

When enabled:

- if no wallet is connected, it shows a connect button
- if a wallet is connected, it shows wallet identity details and a disconnect button

### 14.4 Developer guidance

If a project does not include a `connect` intent, this widget should either:

- remain disabled through `useConnect`, or
- be re-bound in the customization section to another intent or another custom action

This is a core example of the boilerplate philosophy: useful example behavior may ship in the customization section, but it must degrade gracefully when the associated use case is absent.

## 16. Runtime Methods and Developer API

The current global app proxy exposes runtime methods including:

- `app.update()`
- `app.openIntent({ intentKey })`
- `app.buildIntentUrl({ intentKey })`
- `app.buildIntentQr({ intentKey, qrResultType })`
- `app.resetIntent({ intentKey })`
- `app.resetStorage()`

This gives developers a single runtime interface for both direct state mutation and imperative actions.

## 17. Cross-Target Reuse Guidance

### 17.1 Browser HTML

The browser HTML target uses the full shell:

- DOM renderer
- localStorage persistence
- GC browser bundle
- popup or same-window interaction

### 17.2 React reuse

React wrappers should reuse the following conceptual units where possible:

- app state shape
- `defaultIntents` vs `intents`
- `inferFieldsFromJSON(...)`
- intent resolution logic
- validation logic
- result normalization behavior
- connected-wallet normalization

React should replace only the DOM shell and event binding strategy.

### 17.3 NodeJS reuse

NodeJS examples should reuse the following conceptual units where possible:

- state shape
- default/dynamic intent separation
- persistence adapter contract
- intent resolution helpers
- result normalization helpers
- shared business logic that does not depend on DOM state

NodeJS should replace browser storage, browser routing, and DOM rendering only.

## 18. Working Philosophy for Future Boilerplates

The current implementation establishes the following philosophy of work for future boilerplate evolution:

1. Use one global app proxy as the single source of truth.
2. Keep the lifecycle shallow and easy to memorize.
3. Keep use-case logic in the customization section.
4. Keep the runtime generic and reusable.
5. Treat example UI patterns as optional overlays, not mandatory runtime assumptions.
6. Prefer declarative field schemas over scattered field-specific code.
7. Keep transport boundaries small and easy to replace.
8. Preserve end-user clarity even while adding advanced capabilities.
9. Preserve reduced-footprint compatibility in all major design choices.
10. Allow simple cascade updates when `defaultIntents` changes.

## 19. Non-Goals

The current boilerplate is not intended to become:

- a full frontend framework
- a generic component library
- a general-purpose state manager outside the UDC/GCScript context
- an exhaustive catalog of all wallet interaction patterns

Its value lies in being a compact, inspectable, extensible, code-generator-friendly reference shell for GameChanger UDC dapps.

## 20. Acceptance Criteria for the Current Official Boilerplate

An implementation conforming to the current state of this specification should satisfy all of the following:

1. It is deliverable as a single HTML file.
2. It uses one global reactive app proxy created through `createApp({ config })`.
3. It keeps developer customization primarily in the customization section.
4. It keeps the advanced runtime largely use-case-agnostic.
5. It renders forms directly from intent args and options JSON.
6. It persists shared multi-intent state.
7. It accumulates wallet exports into reusable shared state.
8. It supports both URL and QR intent transport.
9. It supports popup and same-window wallet flows.
10. It supports redirect-based and manual pasted wallet response decoding.
11. It supports explicit migration handling when persisted version changes.
12. It keeps the normal user path minimal and end-user focused.
13. It keeps advanced options and debug surfaces hidden unless enabled.
14. It preserves stable DOM ids and classes for downstream styling.
15. It remains compatible with a future reduced-footprint edition.

## 21. Summary

This boilerplate is not just a sample dapp. It is the current reference runtime contract between:

- GCScript intent definitions
- GameChanger code generation tools
- the GC browser transport layer
- end-user-facing Cardano dapps

It now defines not only how generated dapps should look and behave, but also how they should be organized internally: one global app proxy, one narrow customization surface, one generic runtime toolkit, one shared field abstraction model, and one architecture that remains suitable for both readable reference editions and future reduced-footprint editions. Through this structure, the boilerplate supports browser-based, QR-based, popup-based, redirect-based, and air-gapped-adjacent workflows while keeping the developer experience approachable and the runtime small enough to remain practical as a generated artifact.
